## 현재 진행 상황 (2026년 6월)

> 지금 손대고 있는 두 가지 큰 작업이다. 하나는 "데이터를 정리해 디스크를 비우는" 백엔드 작업이고, 다른 하나는 "보기 좋고 쓰기 편하게" 화면을 다듬는 작업이다.

### 1) 데이터베이스 최적화 — morphemes 테이블 샘플링 (완료)

매일 텍스트를 쌓다 보니 형태소 분석 결과를 담는 `morphemes` 테이블이 **약 11.5억 행, 89GB**까지 불어났다. 디스크가 자주 위기였기 때문에, 데이터의 가치를 거의 잃지 않으면서 용량을 줄였다.

- **핵심 아이디어**: 형태소 전체의 **98%가 모두의말뭉치 신문(source_id 6)** 한 곳에서 나온다(약 11.3억). 이 신문 코퍼스는 국립국어원이 1회성으로 적재한 **고정 배경 데이터**라 빈도 집계(`vocab_freq`)와 용례 매핑(`segment_lemma_map`)이 이미 끝나 있다. 따라서 `morphemes`는 이제 "용례 표본 보관소" 역할만 하면 되므로, 신문에서 **단어(lemma·품사)별·하루치당 10개 예시만 남기고** 나머지를 정리했다.
- **건드리지 않는 것**: 신어가 새로 나타나는 **네이버 뉴스·유튜브(댓글·영상)와 모두의말뭉치 구어(source 3·4·5·7)는 전량 보존**한다. 신문이라도 하루 10회 미만 등장하는 **희귀어는 전부 그대로 남는다**(신어 발견에 중요).
- **실제 결과**: 약 **11.5억 행 → 3.4억 행(28GB)**, 약 **70% 감소·디스크 60GB 절약**. `vocab_freq`·`segment_lemma_map`·대시보드 용례·NER 모두 영향 없음을 검증했다.
- **방식**: 인덱스 없는 새 테이블에 "남길 것만" 날짜별로 INSERT한 뒤, 인덱스를 일괄 생성하고 원본과 원자적으로 교체했다. (대용량 테이블에서 `DELETE`는 행마다 인덱스를 갱신해 너무 느렸고 — 하루치 5분 — 인덱스 없는 테이블에 `INSERT`하면 하루치 2초로 약 135배 빨랐다.) 작업 스크립트는 `scripts/sample_morphemes_s6.py`.

### 2) UI/UX 전면 개편 — 대시보드 디자인 시스템 (완료)

Streamlit 대시보드의 디자인 시스템을 전면 개편했다. 기능은 그대로 두되, "처음 보는 사람도 한눈에 이해하고 편하게 쓸 수 있게" 하는 데 초점을 맞췄다.

- **타이포그래피**: 한글 가독성이 좋은 **Pretendard 폰트**를 전체에 적용.
- **컬러 시스템**: 보라색 계열의 **그라데이션 컬러 시스템**으로 통일. 신의미·클러스터·사전 매칭 등 의미가 다른 요소에 일관된 색을 부여해 한눈에 구분되게 했다.
- **입력 컴포넌트 현대화**: 라디오 버튼 → **버튼 그룹**, 체크박스 → **카드** 형태로 바꿔 클릭 대상이 크고 명확해졌다.
- **채팅 UI**: 에이전트 대화 화면을 **말풍선 스타일**로 다듬어, 사용자와 AI의 발화가 분명히 구분된다.
- **데이터 테이블**: 그라데이션 헤더·호버 효과 등 **프리미엄 스타일**을 강화해 표가 읽기 쉬워졌다.
- 이 밖에 모든 페이지 타이틀을 그라데이션 카드로 통일하고, 검색창·버튼·사이드바 스타일을 맞췄으며, AI 검증 결과 카드(신의미 우선 정렬, 정의 관점 배지 등)를 재디자인했다.

## 0. 이 시스템이 하는 일 (한눈에)

매일 한국어 텍스트를 자동으로 모아서, 그 안에서 **사전에 아직 없는 새로운 단어(신어)** 와 **기존 단어의 새로운 의미**를 찾아내는 시스템이다. 찾아낸 결과는 AI(Claude)가 정의문 초안까지 만들어 주고, 웹 화면(대시보드)에서 사람이 확인·검증한다. 사용자는 대시보드에서 직접 찾아보는 것뿐 아니라, **자연어로 묻는 에이전트**("갓생이 요즘 어디서 많이 쓰여?")에게 질문해 답을 받을 수도 있다.

쉽게 비유하면, **어휘 사용을 관찰하는 자동 망원경**이다. 망원경(수집기)이 매일 밤 하늘(인터넷 텍스트)을 찍고, 컴퓨터(분석 파이프라인)가 새로 나타난 별(신어)을 골라내고, AI가 "이 별은 이런 별 같다"고 설명을 붙이고, 연구자가 최종 판단을 한다.

시스템의 관점은 **의미론·코퍼스 언어학** 연구자의 시각에서 설계되었다. 즉, 단순히 단어를 세는 것이 아니라 "이 단어가 어떤 맥락에서 어떤 의미로 쓰이는가", "예전과 지금 의미가 달라졌는가"를 본다.

## 용어 빠른 안내

본문에 자주 나오는 기술 용어를 먼저 한 줄씩 풀어 둔다. (각 장에서 다시 설명하므로 가볍게 훑어도 된다.)

| 용어 | 쉬운 설명 |
|---|---|
| **코퍼스(corpus, 말뭉치)** | 분석 대상이 되는 텍스트의 큰 묶음. "특정 시점 한국어의 표본"이라고 보면 된다. |
| **형태소 분석** | 문장을 단어 단위로 쪼개고 조사·어미를 떼어 사전에 실리는 형태로 바꾸는 작업. (예: "갓생을 살았다" → "갓생", "살다") |
| **표제어(lemma)** | 사전에 올라가는 단어의 기본형. (예: "살았다", "살아서" → 모두 "살다") |
| **임베딩(embedding)** | 문장이나 단어의 "의미"를 컴퓨터가 다룰 수 있게 숫자 목록(벡터)으로 바꾼 것. 의미가 비슷하면 숫자도 가까워진다. |
| **세그먼트(segment)** | 긴 글을 약 300자 단위로 자른 조각. 단어가 실제 쓰인 좁은 문맥만 보기 위한 단위다. |
| **클러스터링(clustering)** | 비슷한 것끼리 자동으로 묶는 작업. 여기서는 비슷한 의미·맥락의 용례를 묶는다. |
| **UMAP** | 1024개나 되는 숫자(고차원 벡터)를 다루기 쉽게 10개 정도(저차원)로 줄여 주는 기법. 의미 구조를 최대한 보존하며 압축한다. |
| **HDBSCAN** | "몇 개로 나눌지" 미리 정하지 않아도 데이터가 빽빽한 곳을 알아서 한 덩어리로 묶어 주는 군집화 기법. 어디에도 안 묶이는 것은 노이즈로 둔다. |
| **LL (로그 가능도, Log-Likelihood)** | "이 단어의 빈도가 통계적으로 의미 있게 늘었는가"를 재는 코퍼스 언어학 표준 지표. 단순 배수보다 안정적이다. |
| **RAG** | AI가 자기 기억이 아니라, 검색해 온 실제 자료를 근거로 답하게 하는 방식(Retrieval-Augmented Generation). |

## 1. 전체 파이프라인 (데이터가 흘러가는 순서)

데이터는 아래 순서대로 한 방향으로 흐른다. 각 단계는 매일 밤 자동으로 차례차례 실행된다.

```
[1] 수집        인터넷에서 새 텍스트를 긁어온다
      ↓
[2] 형태소 분석  문장을 단어 단위로 쪼개고 사전형으로 정리한다
      ↓
[3] 빈도 집계    어떤 단어가 며칠에 몇 번 나왔는지 센다
      ↓
[4] 신어 탐지    사전에 없는 단어 / 갑자기 급증한 단어를 골라낸다
      ↓
[5] 임베딩      문장의 "의미"를 숫자 벡터로 바꾼다
      ↓
[6] 클러스터링   비슷한 의미끼리 묶는다 (한 단어의 여러 의미를 분리)
      ↓
[7] AI 분석     Claude가 의미를 판정하고 정의문 초안을 쓴다
      ↓
[8] 검증·열람    사람이 대시보드에서 확인하고 정정한다
```

각 단계를 아래에서 하나씩 풀어 설명한다.

## 2. 동작 구조 (각 단계 상세)

### 2-1. 수집 (Collection)

매일 정해진 시각에 외부에서 한국어 텍스트를 가져온다. 최종적으로 채택된 출처는 세 갈래다.

- **모두의말뭉치** (국립국어원 제공) — 기반이 되는 대규모 코퍼스(말뭉치). 신문 약 451만 건 + 일상대화 약 2.6만 건. 2019~2024년 자료. 이것은 1회성으로 적재한 "고정 배경 데이터"다.
- **네이버 뉴스** — 공식 검색 API 사용. 매일 새 기사 수집.
- **유튜브 댓글·영상** — 공식 Data API v3 사용. 인기 영상과 그 댓글 수집.

원래 에펨코리아·더쿠 같은 커뮤니티도 후보였으나, 각각 Cloudflare 보안 차단·RSS 잠금으로 **사이트 운영자가 자동 접근을 막아 둔 상태**여서 제외했다. "사이트의 의사에 반하는 우회는 하지 않는다"는 원칙을 지켰다. 이 과정에서 "공개 인터넷 언어 데이터에 대한 접근성이 구조적으로 줄고 있다"는 현상 자체를 연구 배경으로 기록해 두었다.

### 2-2. 형태소 분석 (Morphological Analysis)

수집한 문장은 그대로는 분석할 수 없다. "갓생을 살았다" 같은 문장을 단어 단위로 쪼개고("갓생", "살다"), 조사·어미를 떼고 사전에 실리는 형태(표제어, lemma)로 바꾼다.

- 도구: **Kiwi** (한국어 형태소 분석기)
- 결과는 `morphemes` 테이블에 (단어, 품사, 어느 텍스트에서 나왔는지) 형태로 저장
- 의미를 담는 품사만 남긴다: 일반명사(NNG)·고유명사(NNP)·동사(VV)·형용사(VA). 조사·어미 등은 버린다. (화이트리스트 방식 = 남길 것만 골라 두는 방식)
- Kiwi가 과하게 쪼개는 것을 보정하는 **후처리 규칙**을 직접 짰다. 예: "인공"+"지능" → "인공지능"으로 다시 붙이기, "공부"+"하다" → "공부하다"로 동사화.

### 2-3. 빈도 집계 (Frequency Aggregation)

쪼갠 단어들을 "어떤 단어가, 어떤 매체에서, 며칠에 몇 번 나왔는지" 미리 세어서 `vocab_freq` 테이블에 저장한다.

- 검색할 때마다 전체를 다시 세면 매우 느리므로, **밤에 미리 집계해 두고 낮에는 그것만 꺼내 보는** 방식이다. (정적 집계 = 미리 계산해 두기)
- 여기서 세는 빈도는 기본적으로 "이 단어가 몇 번 등장했는가"이다.

### 2-4. 신어 탐지 (Neologism Detection)

빈도 데이터를 기준으로 "새 단어 후보"를 골라낸다. 두 가지 신호를 쓴다.

- **미등재어**: 사전(우리말샘 등)에 없는 단어 → 신어 후보
- **급등어**: 이번 주 빈도가 지난 4주 대비 갑자기 치솟은 단어 → 떠오르는 신호. 이때 **로그 가능도(LL, Log-Likelihood)** 라는 코퍼스 언어학 표준 지표를 쓴다. (단순히 "몇 배 늘었나"보다 통계적으로 안정적으로 "의미 있는 급등"을 잡아낸다.)

핵심 원칙은 **"수집은 편향 없이, 탐지는 수집된 데이터 안에서"** 이다. 즉, 특정 신어를 노리고 그 단어로 검색해 모으면 데이터가 편향되므로, 수집은 폭넓게(일반 키워드로) 하고 신어 발견은 모인 데이터 안에서만 한다.

### 2-5. 임베딩 (Embedding)

단어의 "의미"를 다루려면 문장을 숫자로 바꿔야 한다. 이를 **임베딩**이라 한다. 의미가 비슷한 문장은 숫자(벡터)도 서로 가까워지므로, 컴퓨터가 "의미가 가깝다/멀다"를 거리로 계산할 수 있게 된다.

- 모델: **bge-m3** (다국어 문장 임베딩 모델, 1024차원 = 문장 하나를 숫자 1024개로 표현)
- **세그먼트 방식**: 긴 텍스트를 300자 단위 조각(세그먼트)으로 잘라, 각 조각마다 의미 벡터를 만든다. 한 텍스트에서 여러 벡터가 나온다.
  - 이 방식의 핵심은 "단어 주변의 좁은 맥락만 본다"는 것. 예전엔 텍스트 앞부분만 임베딩해서 글의 주제(토픽)만 잡히는 한계가 있었는데, 세그먼트 방식으로 단어가 실제 쓰인 좁은 문맥을 보게 됐다.
- `embeddings` 테이블에 (조각 위치, 조각 텍스트, 벡터) 저장
- **매체별 비대칭 보정 샘플링**: 신문이 압도적으로 많아 그대로 두면 신문에 묻힌다. 신문은 1/10만 샘플링하고, 적은 매체는 전부 쓴다.

### 2-6. 클러스터링 (Clustering)

한 단어가 여러 의미로 쓰일 때, 비슷한 의미끼리 자동으로 묶는다(클러스터링 = 비슷한 것끼리 묶기).

- 차원 축소: **UMAP** (1024개나 되는 숫자를 10개 정도로 줄여 다루기 쉽게 만든다)
- 군집화: **HDBSCAN** (밀도 기반. 몇 개로 나눌지 미리 정하지 않아도, 데이터가 빽빽한 곳을 알아서 한 덩어리로 묶는다)
- **매체별로 따로** 클러스터링한다. 문체가 의미보다 먼저 갈리는 경향이 실증으로 확인되었기 때문.

중요한 깨달음 하나: **클러스터 ≠ 의미**. 클러스터는 "맥락"을 묶을 뿐이고, 그것들이 같은 의미인지 다른 의미인지 최종 판단은 다음 단계의 Claude가 한다.

### 2-7. AI 분석 (Claude API)

클러스터별 대표 용례를 Claude에게 보내, 다음을 받는다.

- 의미가 몇 개인지 판정 + 의미별 분류 (여러 클러스터를 한 의미로 통합하거나, 한 클러스터를 나누기도 함)
- 의미별 **정의문 초안** (사전 문체로)
- 대표 용례 선정

출력 형식은 우리말샘 사전 구조를 따르되, 맥락 분포·문법 정보 등을 보강했다. 이 구조 덕분에 시스템은 **부분적으로 RAG**(검색해서 가져온 용례를 근거로 AI가 판단)이자, 전체적으로는 코퍼스 통계 + 벡터 검색 + AI 분석이 결합된 구조다.

### 2-8. 검증·열람 + 에이전트 (Dashboard)

결과는 **Streamlit** 웹 대시보드로 본다. 처음에는 동향·탐색·검증 페이지 위주였으나, 작업이 진행되며 **자연어로 묻는 에이전트를 메인 진입점**으로 끌어올렸다. 현재 페이지 구성과 순서는 다음과 같다.

1. **에이전트와 대화** — 자연어로 질문하면 AI가 알맞은 도구를 호출해 답한다. (메인 진입점, 상세는 4-12절)
2. **어휘 의미 탐색** — 단어 검색 → 빈도, 사전 등재 여부, 용례, 클러스터별 의미
3. **어휘 사용 동향** — 이번 주 떠오른 단어, 매체별 추이
4. **어휘 의미 검증 결과** — Claude 검증 결과 모음 (매체 표시 포함)

자동 분석과 에이전트는 완성됐고, 사람이 직접 정정하는 인터페이스(형태소 정정, 검증 결과 수정, 클러스터 라벨 부여)는 발표 후 단계로 두었다. "자동 분석 + 사람 정정"의 결합이 이 시스템의 지향점이다.

## 3. 전처리 및 수집 전략

### 3-1. 수집 전략 — 가장 중요한 원칙: "수집은 편향 없이, 탐지는 수집된 데이터 안에서"

이 시스템에서 가장 핵심이 되는 수집 원칙이다. 풀어 설명하면 이렇다.

만약 "갓생"이라는 신어가 궁금하다고 해서 "갓생"으로 검색해 글을 모으면, 모인 데이터는 죄다 갓생 이야기뿐이게 된다. 이렇게 모은 데이터는 그 단어가 한국어 전체에서 얼마나 흔한지, 어떤 다른 단어들과 함께 쓰이는지를 전혀 보여주지 못한다. 즉 **신어를 노리고 모으는 순간 데이터가 망가져 일반 코퍼스(언어 표본)로서의 가치를 잃는다.**

그래서 이 시스템은 두 가지를 철저히 분리한다.

- **수집 단계**: 특정 단어를 노리지 않고, 한국어 텍스트를 가능한 한 고르게·폭넓게 모은다.
- **탐지 단계**: 신어 발견은 오로지 이미 모인 데이터 안에서만 한다. 새로 검색해서 끌어오지 않는다.

이 분리 덕분에 모인 데이터는 "특정 시점 한국어의 자연스러운 단면"이 되고, 그 안에서 무엇이 떠오르는지를 객관적으로 잴 수 있다.

### 3-2. 광범위 수집을 실제로 구현한 방법

폭넓게 모으려 해도, 외부 API는 보통 "키워드 없이 전체 최신 글"을 주지 않는다. 그래서 매체마다 우회 설계를 했다.

- **네이버 뉴스**: 네이버의 "전체 최신 뉴스" RSS가 서비스 종료되어, 키워드 검색 API만 남았다. 그래서 **거의 모든 한국 뉴스에 한 번씩은 걸리는 일반 단어 13개**("오늘·어제·발표·공개·정치·경제·사회·문화·스포츠·국제·정부·한국·서울" 등)를 골라, 그 검색 결과의 합집합으로 넓은 커버리지를 만들었다. 한 키워드가 다른 키워드와 같은 기사를 잡으면 중복이 생기는데, 중복이 많다는 것은 오히려 "여러 키워드가 같은 기사를 덮고 있다 = 커버리지가 겹칠 만큼 넓다"는 신호로 해석했다. (단, "서울"처럼 지역 편향을 줄 수 있는 키워드는 나중에 제거)
- **유튜브**: "인기 영상(mostPopular)"을 음악·정치·게임·코미디·연애·라이프스타일 등 **카테고리 7개로 분산**해 가져온다. 한 분야에 쏠리지 않게 하기 위함이다. 영상의 제목·설명과 그 댓글을 함께 수집한다.
- **모두의말뭉치**: 국립국어원이 균형 있게 구축한 코퍼스라, 이것 자체가 "편향 없는 배경 표본" 역할을 한다.

### 3-3. 표본 정합성 — 두 층위로 관리

코퍼스 언어학에서 비교가 의미를 가지려면 표본의 크기·성격이 어느 정도 맞아야 한다. 이 시스템은 정합성을 두 층위로 본다.

- **매체 간 정합성**: 신문이 일상대화의 약 7배 등 매체별 분량이 크게 다르다. 그대로 두면 분석이 신문에 끌려간다. 그래서 임베딩·클러스터링 단계에서 **신문을 1/N로 샘플링**(예: 1/10)해 다른 매체와 균형을 맞춘다.
- **매체 내 정합성**: 같은 매체 안에서도 쏠림이 있다.
  - 유튜브 댓글이 인기 채널 하나에 집중 → **채널당 댓글 상한**을 둔다.
  - 네이버 뉴스는 같은 기사가 여러 언론사로 재배포되어 중복 → **본문 앞부분(예: 앞 80자) 기준 중복 제거**.
  - 모두의말뭉치 신문은 2020~2022년 코로나·정치 보도에 시기가 편중 → 이 편중을 분석 시 의식하고 자가 인식 항목으로 기록.

### 3-4. 수집 시 지킨 윤리·법적 원칙

- **사이트 의사를 존중한다.** 에펨코리아는 Cloudflare 보안 차단, 더쿠는 RSS 잠금으로 자동 접근을 명시적으로 막아 두었다. 기술적으로 우회할 수도 있었지만 "운영자의 의사에 반하는 우회는 하지 않는다"는 원칙으로 둘 다 제외했다.
- **공식 무료 경로만 사용한다.** 네이버 검색 API(하루 25,000 호출 한도), 유튜브 Data API v3(일일 quota 10,000 units 중 약 5%만 사용)처럼 제공자가 허용한 공식 API만 쓴다.
- **비용·한도 가드.** 호출량이 한도를 넘지 않도록 보수적으로 운영한다.
- 이 과정에서 "2023년 이후 공개 커뮤니티·SNS의 자동 접근 제한이 빠르게 확산되고 있다"는 현상 자체를 연구 배경으로 기록했다. (X 유료화 → 에펨 Cloudflare → 더쿠 RSS 잠금 → 네이버 RSS 종료)

### 3-5. 데이터 보존 우선 원칙 — "잃기보다 크게 저장하고 거른다"

전처리에서 일관되게 지킨 큰 방향이다. **노이즈가 섞이더라도 일단 넓게 저장하고, 거르기는 나중 단계(쿼리·후보 추출·사람 검토)에서 한다.** 데이터를 미리 버리면 되살릴 수 없지만, 저장해 두면 언제든 거를 수 있기 때문이다.

예: "0.08%p" 같은 퍼센트포인트 표기에서 Kiwi가 'p'를 외국어 토큰으로 잘못 잡아 신문에서 하루 수천 번씩 등장하는 노이즈가 생겼다. 하지만 'AI'·'GDP' 같은 알파벳 약어는 다른 분석에 쓸모가 있으므로, **데이터는 그대로 두고 신어 후보 추출 단계에서만 필터링**하기로 했다.

### 3-6. 형태소 전처리 — 한국어에 맞춘 정교한 설계

수집한 문장을 분석 가능한 단어 단위로 바꾸는 단계로, 한국어 형태론 지식이 많이 들어갔다.

- **분석기**: Kiwi. 사용자 사전은 일부러 쓰지 않는다(어떤 단어를 사전에 넣을지 계속 관리하는 부담을 피하기 위함).
- **의미를 담는 품사만 남긴다(화이트리스트)**: 일반명사(NNG)·고유명사(NNP)·동사(VV)·형용사(VA)만 저장하고, 조사·어미 등 나머지는 버린다. 블랙리스트(버릴 것만 지정)보다 안전하다 — 예상 못 한 태그가 새어 나오지 않기 때문.
- **후처리 규칙(merge_tokens)** — Kiwi가 과하게 쪼갠 것을 다시 합친다.
  1. 위치가 붙어 있는 명사+명사 → 복합명사로 결합 ("인공"+"지능" → "인공지능"). 띄어쓰기가 있으면 위치가 끊기므로 합치지 않는다.
  2. 명사+동사화 접미사(XSV) → 동사 ("공부"+"하" → "공부하다").
  3. 명사+형용사화 접미사(XSA) → 형용사 ("사랑"+"스럽" → "사랑스럽다").
  4. 동사/형용사+연결어미+보조용언 → 보조용언 결합 ("살"+"아"+"보" → "살아보다"). 이때 원문에서 직접 잘라내 표기 왜곡을 막는다.
- **표제어 결정 보류 전략**: "갓생"(명사)과 "갓생하다"(동사)를 둘 다 따로 빈도 집계해 두고, 어느 쪽을 사전 표제어로 삼을지는 나중 AI 분석에서 용례·빈도를 보고 판단한다.

### 3-7. 어절 복원 — 형태소 분석기의 사전 한계를 보완하는 두 번째 트랙

형태소 분석기는 사전에 기반하므로, 사전에 없는 신어를 통째로 쪼개 버린다("갓생" → "갓"+"생"). 특히 SNS·댓글의 신조어가 신문에서는 잡히지 않는다. 그래서 별도의 **어절 복원 트랙**을 자체 개발했다. (어절 = 띄어쓰기로 나뉘는 단위)

- Kiwi가 쪼갠 토큰들을 띄어쓰기(어절) 단위로 다시 붙이고, 조사·어미를 떼어 신어 후보를 만든다.
- 댓글 특유의 비표준 표기(붙여쓰기)와 강조부사 결합을 처리한다.
- 이 알고리즘은 v1→v4로 진화하며 다듬어졌다. 고유명사 제외, 서술격조사("~이다")의 "이" 노이즈 제거, 동사 어간 별도 추출 등을 거쳐, 결국 **"개웃기다·킹받다·빵터지다"** 같은 합성 신어까지 잡게 됐다.
- **노이즈는 두는 쪽**을 택했다(3-5 원칙과 일관). 강조부사는 계속 새로 생기므로("꿀잼·극혐·쌉가능") 화이트리스트로 관리하기 어렵고, 노이즈 비율 약 15~20%는 사람 검토로 거를 수 있다고 판단했다. 알려진 노이즈 패턴("안/잘/못+동사", "명사+드리다" 등)은 검토 시 인지하도록 기록해 두었다.

### 3-8. 기타 전처리 디테일

- **HTML 정리**: 네이버 API 응답의 `<b>` 강조 태그와 HTML 엔티티를 정규식 + unescape로 제거.
- **언어 필터**: 영어가 지배적인 댓글은 한국어 비중 30% 미만이면 임베딩 대상에서 제외.
- **표제어 정규화(headword_norm)**: 사전 매칭을 위해 하이픈("필요-하다"→"필요하다")·캐럿("방역^수칙"→"방역수칙")·공백을 제거. 이 정규화 하나로 사전 매칭률이 눈에 띄게 올랐다.
- **세그먼트 분할**: 임베딩 직전, 긴 텍스트를 300자 단위로 잘라 단어 주변 좁은 맥락만 보게 한다(상세는 4장).

### 3-9. 데이터 엔지니어링 패턴 (전처리·집계의 안전장치)

- **증분 ETL**: 매일 새로 들어온 것만 처리해 부담을 줄인다. (ETL = 추출·변환·적재의 데이터 처리 흐름)
- **멱등성**: 같은 작업을 두 번 돌려도 결과가 망가지지 않게 설계한다(DELETE 후 INSERT, 또는 ON CONFLICT). 빈도 집계에서 같은 기간을 두 번 더하면 수치가 2배가 되는 함정을 이 패턴으로 막는다.
- **Backfill 윈도우**: 늦게 들어온 데이터를 위해 최근 7일치를 다시 처리한다.
- **미리 계산(요약 테이블)**: 9천만 행짜리 빈도 테이블을 매번 전수 집계하면 느리므로, `vocab_lemma_summary` 같은 요약 테이블을 미리 만들어 둔다(전수 집계가 30분 → 요약 테이블 조회는 초 단위).
- **중복 방지**: DB에 UNIQUE 제약 + `ON CONFLICT DO NOTHING`을 걸어 같은 글이 두 번 저장되지 않게 한다.

## 4. 분석 전략

### 4-1. 분석은 두 층위로 이루어진다

이 시스템의 분석은 성격이 다른 두 층위가 맞물려 돌아간다.

- **빈도·통계 층위 (코퍼스 언어학)**: "무엇이 얼마나 쓰이고, 무엇이 떠오르는가"를 객관 수치로 잡는다. 빈도 집계, 키워드 비교(LL), 시계열 추이가 여기 속한다.
- **의미 층위 (분포 의미론)**: "그 단어가 어떤 의미로 쓰이는가"를 맥락 분포로 잡는다. 임베딩, 클러스터링, AI 의미 분석이 여기 속한다. (분포 의미론 = "단어의 의미는 함께 쓰이는 말들로 드러난다"는 관점)

신어 발견은 통계 층위에서 시작해(무엇이 떠올랐는가), 의미 층위로 넘어가(그게 무슨 뜻인가) 완성된다.

### 4-2. 신어를 골라내는 두 가지 통계 신호

빈도 데이터에서 신어 후보를 뽑는 기준은 두 갈래다.

- **미등재어 신호**: 사전(우리말샘 등)에 없는 단어다. 매칭 시 세부 규칙을 둔다.
  - 사전 표제어 존재 자체로 판정하고(품사 결합은 안 함 — 사전의 품사 정보가 빈 항목이 35%나 되어 결합하면 누락이 생김), 비교 대상은 `sense_type='일반어'`만 쓴다(방언·북한어·옛말로만 등재된 것은 "현대 표준어로 등재됨"으로 보기 어렵고, 옛말이 새 의미로 부활한 경우는 신어로 잡혀야 하므로).
  - vocab_freq 쪽 노이즈 필터: 품사는 NNG/VV/VA, 글자 길이 2 이상, 한글만(`^[가-힣]+$`), 누적 빈도 100 이상.
- **급등 신호**: 이번 주 빈도가 지난 4주 평균 대비 갑자기 치솟은 단어다. 여기서 **로그 가능도(LL, Log-Likelihood)** 라는 코퍼스 언어학 표준 지표를 쓴다. 단순 배수보다 통계적으로 안정적으로 "유의미한 급등"을 잡아낸다.

후보의 빈도 분포는 전형적인 멱법칙(소수의 고빈도어 + 다수의 저빈도어)을 따르며, 검토는 점수(빈도) 상위 순으로 한다.

### 4-3. 신어의 성격을 시간 추이로 분류한다

단순 빈도만이 아니라 **시간에 따른 빈도 패턴**으로 신어의 성격을 나눌 수 있음을 발견했다.

- **사건어형**: 특정 시기에 집중되었다가 이후 사그라드는 말. 예: 코로나기(2020~2021)의 "확산세·자가격리·방역수칙·재난지원금".
- **정착어형**: 등장 후 오히려 꾸준히 늘어나는 말. 예: "커머스·헬스케어·원스톱".

이 구분은 "이 단어가 일시적 유행인가, 사전에 올릴 만큼 정착했는가"를 판단하는 메타정보가 된다.

### 4-4. 개체명(고유명사)을 따로 다룬다

신어 후보에는 인명·기관명·작품명 같은 고유명사가 대량으로 섞여 들어와 가장 큰 노이즈가 된다. 이를 처리하기 위해 별도의 **개체명 인식(NER, Named Entity Recognition = 사람·기관·장소 등의 고유명사를 자동으로 식별하는 기술)** 도구를 도입했다.

- 모델: KoELECTRA-NER 계열(한국어 표준 KLUE 데이터셋 기반). 인물(PS)·기관(OG)·장소(LC)·날짜·수량·시간 등을 분류한다.
- 효과: "츠키·아이브" 같은 고유명사를 일반어 신어 목록에서 빼서 별도 "인물·단체" 섹션으로 옮긴다. 그러면 "촉법·환승연애·땡기다" 같은 진짜 발견이 부각된다.
- 다만 고유명사 트랙 자체도 버리지 않는다. 정치인·연예인·게임 캐릭터 등의 등장 추세는 그 자체로 가치 있는 데이터이므로 별도 트랙으로 활용한다.
- 한계도 명시한다: 정부 기관명을 못 잡거나 일부를 잘못 분류하는 등 약 7%의 오차가 남는다.

### 4-5. 의미를 다루는 핵심 — 분포 의미론과 세그먼트 임베딩

이 시스템이 의미를 보는 관점의 토대는 **분포 의미론**이다. "단어의 의미는 그 주변에 함께 나오는 말들로 드러난다"는 생각이다.

- 그래서 단어가 실제로 쓰인 **좁은 맥락**을 봐야 한다. 초기에는 텍스트 앞부분(앞 512자)만 임베딩했는데, 그러면 글 전체의 주제(토픽)만 잡히고 단어 자체의 의미는 흐려지는 한계가 있었다.
- 이를 해결한 것이 **세그먼트 임베딩**이다. 긴 텍스트를 300자 단위로 잘라 각 조각(세그먼트)의 의미 벡터를 만들고, 특정 단어를 분석할 때는 **그 단어가 등장한 조각의 벡터만** 골라 쓴다. "단어 좌우의 좁은 윈도우만 본다"는 정신을 구현한 것이다.
- 단어와 조각을 빠르게 연결하기 위해 **세그먼트↔단어 매핑 테이블**(약 1억 4천만 행)을 미리 만들어 둔다. 검색할 때마다 즉석에서 찾는 대신, 사실로 기록해 둔 매핑을 조회한다.

### 4-6. 비슷한 의미끼리 묶기 — 클러스터링, 그리고 "클러스터 ≠ 의미"

- **차원 축소(UMAP)**: 1024차원 벡터를 다루기 쉽게 줄인다. (수백~천 개의 숫자를 의미 구조를 보존하며 10개 안팎으로 압축)
- **군집화(HDBSCAN)**: 밀도 기반이라 "몇 개로 나눌지" 미리 정하지 않아도 자동으로 묶고, 어디에도 안 묶이는 것은 노이즈로 둔다.
- **매체별로 따로 묶는다**: 문체(사용역)가 의미보다 먼저 갈리는 경향이 실증으로 확인되었기 때문이다(뉴스 기준으로 검색하면 일상대화에 같은 의미가 있어도 뉴스만 묶인다). 그래서 단어×매체 단위로 클러스터링한다.

여기서 가장 중요한 깨달음: **클러스터는 "의미"가 아니라 "맥락"이다.** 같은 의미라도 맥락이 다르면 여러 클러스터로 갈리고, 반대로 다른 의미가 한 클러스터에 섞이기도 한다. 클러스터를 의미로 통합하거나 분리하는 최종 판단은 다음 단계의 AI가 한다.

### 4-7. AI 의미 분석 — 클러스터를 의미로 번역한다

클러스터별 대표 용례를 Claude에게 넘겨, 사람 사전편찬자가 할 법한 판단을 받는다.

- 의미가 몇 개인지 판정하고, 클러스터들을 의미 단위로 통합·분리한다.
- 의미별 **정의문 초안**을 사전 문체로 쓴다.
- 의미별 대표 용례를 고른다.
- 출력은 우리말샘 사전 구조를 따르되, 어떤 클러스터들을 합쳤는지·맥락 분포·문법 정보 등을 보강한다.

이 구조는 **부분적으로 RAG**(Retrieval-Augmented Generation = 검색해 온 자료를 근거로 생성)다. 즉 모델이 자기 기억으로 답하는 게 아니라, 코퍼스에서 검색해 온 실제 용례를 근거로 판단한다. 다만 시스템 전체는 RAG보다 넓어서, 코퍼스 통계(빈도·LL)·임베딩·클러스터링 등 비-RAG 요소가 결합되어 있다.

### 4-8. 정의문을 쓰는 관점 — 어휘 유형마다 정의 방식이 달라야 한다

이 시스템에서 정의문 생성은 "AI에게 뜻풀이를 시킨다"는 단순 작업이 아니다. **단어의 유형에 따라 좋은 정의문의 방식 자체가 다르다**는 사전학적 통찰을 전략으로 옮긴 부분이다. 이 절을 특히 자세히 둔다.

#### 왜 하나의 정의 방식으로는 안 되는가

전통적인 단어 정의 공식은 "상위 개념 + 변별 특성"이다(국제 표준 ISO 704의 방식). 예를 들어 "참새: 몸집이 작은(변별 특성) 새(상위 개념)"처럼 쓴다. 이 방식은 사물·개념을 가리키는 명사에는 잘 맞는다.

그러나 신어·변화어를 실제로 들여다보면, 이 공식이 안 맞는 유형이 수두룩하다.

- **외래어 합성어**(예: "OO템", "워라밸")는 상위 개념을 따지기보다 **어떤 말들이 어떻게 결합했는지(어원·구성)** 를 풀어야 뜻이 산다.
- **강조 부사·접두 요소**(예: "개~", "핵~", "찐~")는 가리키는 사물이 없다. **어떤 맥락에서 어떤 강조 효과로 쓰이는지**를 기술해야 한다.
- **기존 단어의 새 의미**(예: "긁다"가 '카드 결제하다', '자극하다'로)는 정의를 새로 쓰는 게 아니라 **원래 의미에서 어떻게 갈라져 나왔는지**를 보여야 한다.
- **줄임말**(예: "갑분싸")은 **무엇의 준말인지 복원**하는 것이 정의의 핵심이다.

즉, 외래어 합성에는 ISO 704식 단어 정의가 부자연스럽고, 오히려 **어원학 + 말뭉치 의미론 + 한국 사전학 전통**의 방식이 더 자연스럽다. 이런 인식에서 출발해, 단어 유형에 따라 정의 관점을 골라 쓰는 전략을 세웠다.

#### 정의 관점 도구함 — 8가지 관점

그래서 AI에게 단 하나의 정의 공식을 강제하지 않고, **여러 정의 관점을 "도구함"처럼 갖춰 두고 단어 유형에 맞는 것을 골라 쓰게** 했다. 8가지 관점은 각각 언어학의 다른 전통에서 가져왔다.

1. **상위 개념 + 변별 특성** (ISO 704) — 전형적 사물·개념 명사에 적합.
2. **어원·구성 요소 분석** (형태론·어원학) — 합성어·외래어·파생어에 적합. 어떤 요소가 어떻게 결합했는지 푼다.
3. **프레임·상황 도식** (Fillmore의 프레임 의미론) — 특정 상황·행위와 묶여 쓰이는 말에 적합.
4. **원형 + 변이형** (Rosch의 원형 이론) — 전형적 사례를 중심으로 의미를 잡는 말에 적합.
5. **전형적 결합·용법** (Sinclair·Hanks의 말뭉치 의미론) — 실제로 어떤 단어들과 함께 쓰이는지(연어)로 뜻을 기술.
6. **의미 관계 분석** — 상위어·하위어·반의어·유의어·부분전체 관계 속에서 위치를 잡음.
7. **사회·시대·집단 맥락** — 어떤 집단이 언제부터 어떤 함의로 쓰는지.
8. **비유·은유 분석** — 비유적 전이로 생긴 의미에 적합.

#### 강제 라우팅이 아니라 자유 선택 + 사용 관점 기록

설계에서 의도적으로 피한 함정이 있다. "이 단어는 외래어니까 무조건 2번 관점" 식으로 **기계적으로 강제 배정(라우팅)하지 않는다.** 실제 신어는 형태와 의미가 거의 항상 함께 작동해서 "형태 신어 / 의미 신어"로 깔끔히 나뉘지 않고, 한 단어에 여러 관점이 동시에 필요한 경우가 많기 때문이다.

그래서 택한 방식은 이렇다.

- AI가 그 단어의 용례를 보고 **적합한 관점을 스스로 골라** 정의문을 쓴다.
- 단, 어떤 관점을 참고했는지를 `perspectives_used`라는 메타 필드에 **기록**해 둔다.
- 모든 단어에 8개 관점의 정의를 다 만드는 것은 과도하므로 하지 않는다. 쓴 관점만 남긴다.

이렇게 하면 유형별 유연성(단어에 맞는 정의 방식)과 사후 검증 가능성(왜 이렇게 정의했는지 추적)을 동시에 얻는다.

#### 어휘 변화 유형 7분류 — 코퍼스 언어학에 맞춘 라벨

정의 관점과 짝을 이루어, 검증 결과에 붙이는 **어휘 변화 유형 분류**도 코퍼스 언어학의 어휘 변화 틀에 맞게 다시 설계했다. 초기의 단순한 4분류를 버리고 다음 7가지로 정교화했다.

신조어, 외래어·외국어, 의미 변화, 합성, 파생, 줄임말, 노이즈.

여기서 "합성"의 정의에도 관점이 들어가 있다. 외래어와 한국어가 섞인 혼종 결합(예: 외래어 어근 + 한국어 접사)도 **구성 원리를 중심으로 "합성"으로 분류**한다. 이는 단어의 출신(외래어냐 고유어냐)이 아니라 만들어진 방식(어떻게 결합했느냐)을 기준으로 본다는 입장이다.

#### 의미 기술은 정의문 한 줄이 아니다

마지막으로, 이 시스템은 의미 기술을 정의문 한 문장으로 끝내지 않는다. **정의문 + 용례 + 사용역(어떤 매체에서 어떤 결로 쓰이는가)** 의 묶음으로 본다. AI가 쓴 정의문이 실제 코퍼스 분포에 부합하는지를 항상 함께 점검하며, 이 점에서 우리말샘 같은 실제 사전 집필 방식과 정합한다(우리말샘의 12개 기술 항목 중 표제어·품사·원어·범주·뜻풀이·용례 등 5~7개를 자동 생성으로 보조할 수 있다).

### 4-9. 사전을 검증한다 — 다중 사전 비교와 의미 번호 검증

이 시스템의 산출물은 단순 신어 목록이 아니라, **기존 사전을 검증하고 갱신 정책을 결정하는 도구**를 지향한다.

- **다중 사전 비교**: 우리말샘·표준국어대사전·한국어기초사전 세 사전을 모두 적재했다(합계 약 174만 의미). 사전마다 의미 번호 체계가 달라, 같은 단어를 세 사전이 어떻게 다르게 나누는지 비교할 수 있다.
- **의미 번호 검증(sense_validation)**: 사전이 나눠 놓은 의미 구분이 실제 사용 데이터에 부합하는지 점검한다. 사전에 있는 의미가 실제로는 안 쓰이거나, 사전에 없는 새 의미가 데이터에 나타나는 경우를 잡아낸다.

### 4-10. 통시적 의미 변화 추적 (다음 단계 지향)

분포 의미론과 짝을 이루는 **통시 의미론**의 관점도 설계에 들어 있다. 같은 단어라도 시기에 따라 의미가 달라진다는 것이다(통시 = 시간 흐름에 따른 변화). 예를 들어 "국민배당금"은 예전 정치 맥락과 최근의 새 맥락에서 다른 결로 쓰인다.

현재는 모든 시기 용례를 한꺼번에 클러스터링하지만, 다음 단계로 **시기 창(time window)을 나눠 같은 단어를 시기별로 따로 검증**해 "이 의미는 언제부터 생겼는가"에 직접 답하는 방향을 지향한다. 사전편찬자가 진짜 알고 싶어 하는 질문이기 때문이다.

### 4-11. 자가 인식 — 한계를 정직하게 기록한다

분석의 한계를 숨기지 않고 메타정보로 남기는 것이 이 시스템의 특징이자 강점이다.

- 형태소 분석기의 미등재어 분할 한계(긴 신어가 짧은 등재어로 쪼개짐, 예: "어뷰징"→"뷰징").
- 동사·형용사 활용형이 일부(약 7%) 잘못 박히는 문제.
- 클러스터링이 단어 의미가 아니라 글 전체의 주제 프레임을 잡는 한계.
- 임베딩이 좁은 맥락보다 발화 전체의 토픽(예: 특정 인물 밈)을 반영하는 한계.
- 모두의말뭉치의 시기 편중(2020~2022 코로나·정치 보도).
- NER의 약 7% 오차.

특히 AI 검증 단계에서 Claude가 이런 한계를 스스로 진단하도록 설계되어 있다(예: "임베딩이 표제어 의미보다 발화 전체 토픽을 반영해 의미가 아니라 토픽 단위로 갈렸다"고 자가 진단). 이 정직성이 "자동 분석 + 사람 정정"이라는 다음 단계로 자연스럽게 이어진다.

### 4-12. 언어 관측 에이전트 — 자연어로 묻고 도구로 답한다 (Agentic RAG)

이 시스템의 분석 기능들(빈도 조회·추이·검증 등)은 원래 페이지마다 흩어져 있었다. 이것들을 **하나의 자연어 대화 창구로 묶은 것이 에이전트**다. 사용자가 "갓생 요즘 어디서 많이 쓰여?", "지금까지 관측한 거 정리해줘" 같은 말을 던지면, AI가 알맞은 도구를 골라 호출하고 결과를 종합해 답한다. 대시보드에서 이 에이전트를 **메인 진입점(1번 페이지)** 으로 두었다.

#### 어떻게 동작하나 — 도구 호출(tool-use) 방식

핵심은 **AI가 직접 DB를 헤집는 게 아니라, 미리 만들어 둔 "도구 함수"를 골라 부른다**는 점이다. 사람이 "이 질문엔 빈도 조회 도구가 맞겠다"고 판단하듯, AI가 질문을 보고 어떤 도구를 쓸지 스스로 정한다.

- 모델: Claude(opus 계열) + **tool-use(함수 호출)** 기능.
- `agent_tools.py`에 도구 함수들을 정의하고, `agent_loop.py`의 `run_agent()`가 대화 루프를 돌린다.
- 루프는 "질문 → AI가 도구 선택 → 도구 실행 → 결과를 AI에게 전달 → 필요하면 또 다른 도구 호출 → 최종 답변" 순으로 진행되며, 무한 반복을 막기 위해 최대 호출 횟수(10회) 제한을 둔다.
- 진행 과정을 단계별로 화면에 흘려보내(`yield`) 사용자가 "지금 어떤 도구를 부르는 중"인지 볼 수 있다.

#### 도구 5종

사람 연구자가 던지는 네 가지 질문(이 단어 뜻이 뭐야 / 어떻게 쓰여 / 언제·어디서 쓰여 / 사전마다 어떻게 다른가)에서 출발해, 도구를 다음 5종으로 정리했다.

| 도구 | 하는 일 |
|---|---|
| `lookup_word(lemma, pos)` | 한 단어의 모든 정보(빈도·사전 등재 여부·신어 후보 여부·검증 결과)를 한 번에 모아 준다. |
| `freq_by_source(lemma, pos, days)` | 매체·시기별 빈도 추이와 정점(peak)을 보여 준다. |
| `find_emerging_words(days)` | 최근 떠오른 단어들과 매체별 LL 키워드를 뽑아 준다. |
| `validate_sense(lemma, pos, source_id)` | 그 자리에서 클러스터링 + Claude 검증을 실행한다(2~6분 소요). |
| `system_overview()` | 시스템이 지금까지 모으고 발견한 것을 종합 요약한다. |

설계에서 주목할 점 두 가지. (1) `validate_sense`는 응답에 몇 분이 걸리지만 "조금 기다리게 하면 된다"는 판단으로 살려 두어, 에이전트가 즉답뿐 아니라 무거운 분석까지 시킬 수 있게 했다. (2) `system_overview`는 처음엔 9천만 행짜리 빈도 테이블 전수 집계를 포함했다가 너무 느려, 코퍼스 텍스트·댓글 통계만 남기는 쪽으로 줄였다(전처리 장의 "미리 계산" 원칙과 같은 맥락).

#### 채팅 UI

Streamlit의 채팅 컴포넌트로 대화창을 만들고, 사이드바에 예시 질문과 "대화 새로 시작" 버튼을 두었다. 특히 **도구 호출 트레이스**(AI가 어떤 도구를 어떤 인자로 불렀는지)를 답변 위에 펼침 형태로 보여 준다. 답이 어떻게 나왔는지 과정을 투명하게 드러내, 4-11절의 "정직성" 원칙과 이어진다.

#### 시스템 정체성에서의 위치

이 에이전트는 **Agentic RAG**(스스로 도구를 골라 자료를 검색해 답하는 방식)에 해당한다. 앞서 본 claude_analyzer(검증)·주간 보고서 코멘트와 함께 시스템의 RAG 계열 요소를 이루고, 임베딩·클러스터링·코퍼스 통계 같은 비-RAG 요소와 결합한다. 그래서 시스템 전체의 정체성은 **"코퍼스 언어학 + 벡터 검색 + Agentic RAG의 결합"** 으로 정리된다.

## 5. 기술 스택·인프라 요약

- **DB**: PostgreSQL + pgvector(벡터 검색 확장), Docker 컨테이너로 운영
- **형태소 분석**: Kiwi (kiwipiepy)
- **임베딩**: bge-m3, GPU(RTX 5060 Ti)로 대량 처리
- **클러스터링**: UMAP + HDBSCAN (sklearn 계열)
- **AI 분석·에이전트**: Claude API (opus 계열) — 의미 검증(claude_analyzer)과 tool-use 기반 에이전트(`agent_tools.py`의 도구 5종 + `agent_loop.py`의 run_agent 루프)
- **대시보드**: Streamlit (에이전트 채팅 페이지 포함) — 2026년 6월 디자인 시스템 전면 개편(Pretendard 폰트, 보라색 그라데이션 컬러 시스템, 카드형 입력 컴포넌트, 말풍선 채팅 UI 등)
- **자동화**: cron 스케줄러 + `run_pipeline.sh` 래퍼 스크립트
- **외부 공개**: Cloudflare Tunnel → `lang-observatory.com`

운영 환경은 처음 **AWS EC2**(t3.large → m5.xlarge로 업그레이드, EBS 30→200GB로 확장)에서 시작했으나, 비용·GPU 필요성 때문에 **로컬 PC(WSL/Ubuntu + GPU)** 로 마이그레이션했다. 현재 PC가 매일 밤 자동으로 깨어나 파이프라인을 돌리고 새벽에 절전된다.

주요 DB 테이블(현재 기준):

| 테이블 | 내용 | 규모 |
|---|---|---|
| texts / comments | 수집 원문 | 수백만 건 |
| morphemes | 형태소 분석 결과 | 약 3.4억 행 (약 28GB) — *신문 코퍼스를 단어·일별 10개 예시로 샘플링해 11.5억 행에서 축소* |
| vocab_freq | 단어별 일별 빈도 | 약 9천만 행 |
| embeddings | 세그먼트 의미 벡터 (1024차원) | 약 312만 행 |
| segment_lemma_map | 세그먼트↔단어 매핑 | 약 1억 4,360만 행 |
| neologism_candidates | 신어 후보 | 약 5만 건 |
| usage_clusters / members | 클러스터링 결과 | — |
| 사전 테이블(3종 통합) | 사전 의미(sense) | 약 174만 건 |

> 참고: `embeddings`·`segment_lemma_map`은 매일 새 데이터가 들어와 행 수가 계속 늘어난다. 위 수치는 2026년 6월 14일 파이프라인 실행 기준이다.

## 6. 실제 세팅 과정 (재현 가이드)

> 이 시스템을 처음부터 다시 세운다고 할 때의 큰 순서다. 개발 경험이 없어도 따라갈 수 있도록 단계별로 정리했다. 명령어는 핵심만 예시로 든다.

### 6단계 1. 서버·DB 준비

1. **운영 환경 선택**
   - 클라우드(AWS EC2)로 시작하거나, GPU가 달린 로컬 PC(Windows + WSL/Ubuntu)로 시작한다.
   - 클라우드는 켜고 끄기 쉽지만 GPU 임베딩 비용이 들고, 로컬 PC는 전기·관리가 필요하지만 GPU를 마음껏 쓴다. (이 프로젝트는 결국 로컬 PC로 옮겼다.)

2. **PostgreSQL + pgvector를 Docker로 띄운다**
   - `docker-compose.yml`에 `pgvector/pgvector:pg16` 이미지를 적고, 데이터가 사라지지 않게 볼륨을 연결, 재부팅 시 자동 복구(`restart: unless-stopped`)를 켠다.
   - `docker compose up -d`로 실행하고, 접속해서 `CREATE EXTENSION vector;`로 벡터 기능을 켠다.

3. **비밀정보 관리**
   - DB 비밀번호·API 키는 `.env` 파일 한 곳에 모으고 본인만 읽도록 잠근다(`chmod 600`).

### 6단계 2. 스키마(테이블) 만들기

- 설계해 둔 SQL로 테이블들을 한 번에 생성한다 (sources, texts, comments, morphemes, embeddings, vocab_freq, neologism_candidates, 클러스터·사전·정의·보고서 테이블 등).
- `sources` 테이블에 출처 목록(모두의말뭉치·네이버·유튜브 등)을 먼저 입력해 둔다.

### 6단계 3. Python 환경 준비

- 가상환경을 만든다: `python3 -m venv venv` → `source venv/bin/activate`
- 필요한 패키지 설치: `psycopg2-binary`(DB 연결), `kiwipiepy`(형태소 분석), `sentence-transformers`(임베딩), `umap-learn`·`hdbscan`(클러스터링), `streamlit`(대시보드) 등.
- 프로젝트 루트에서 `src/` 모듈을 쓰려면 `PYTHONPATH=src python3 스크립트.py` 형태로 실행한다.

### 6단계 4. 기반 코퍼스 적재 (1회성)

- 모두의말뭉치 원본 JSON을 서버로 옮기고, 파싱 스크립트(`load_newspaper.py`, `load_dialogue.py`)로 `texts` 테이블에 넣는다. (약 453만 건)
- 대용량 작업이므로 `tmux` 세션 안에서 돌려, 접속이 끊겨도 계속 진행되게 한다.

### 6단계 5. 형태소 분석 → 빈도 집계

- 형태소 분석 스크립트(`src/analyze_morphemes_tf.py`)를 돌려 `morphemes`를 채운다. 매우 오래 걸리므로(수 시간~수십 시간) 반드시 백그라운드/tmux에서.
  - 빠른 적재를 위해 INSERT 대신 **COPY 벌크 로드**를 쓰고, 적재 중엔 인덱스를 빼두었다가 끝나고 다시 만든다.
- 빈도 집계 스크립트(`build_vocab_freq.py`)로 `vocab_freq`를 채운다. 월 단위로 나눠 처리하면 중간 실패에 강하다.

### 6단계 6. 사전 적재

- 우리말샘·표준국어대사전·한국어기초사전 원본을 각각의 파서로 변환해 사전 테이블에 넣는다. 표제어 정규화(하이픈·공백·캐럿 처리)를 거친다.

### 6단계 7. 임베딩 (GPU 권장)

- 세그먼트 단위 임베딩 스크립트(`src/analyzers/embed_texts_segmented.py`)를 GPU로 돌려 `embeddings`를 채운다.
- 매체별 샘플링 비율을 코드에 설정한다(신문 1/10 등).
- CPU로는 수십 일, GPU로는 수 시간 규모다.
- 끝난 뒤 세그먼트↔단어 매핑(`src/build_segment_lemma_map.py`)을 만들고, 벡터 검색용 인덱스(ivfflat)를 생성한다.

### 6단계 8. 분석 파이프라인 + 에이전트 연결

- 클러스터링(`src/analyzers/cluster_usage.py`)과 Claude 분석(`src/analyzers/claude_analyzer.py`)을 붙인다.
- 신어 후보 중 미처리 상태인 것을 자동으로 분석하도록 한다.
- 에이전트를 붙인다: 도구 함수 5종(`src/analyzers/agent_tools.py`)을 정의하고, tool-use 대화 루프(`src/analyzers/agent_loop.py`의 `run_agent()`)를 만든 뒤, Streamlit 채팅 페이지(1번 "에이전트와 대화")에 연결한다. 도구 호출 트레이스를 화면에 펼쳐 보이게 한다.

### 6단계 9. 자동화 (cron)

- 각 단계를 `scripts/run_pipeline.sh {naver|youtube|morphemes|embed|segment_map|refresh|eojeol|report|all}` 한 스크립트로 호출하게 만든다(가상환경 활성화·로그 기록 포함).
- cron에 야간 시간표로 등록한다. 현재 운영 시간표:

```
22:00  DB 백업
23:30  네이버 뉴스 수집
23:35  유튜브 수집
00:00  형태소 분석
01:00  빈도 재집계
02:00  임베딩
02:30  세그먼트-단어 매핑
03:00  어절 복원 신어 탐지
05:00 (월) 주간 보고서
```

- 로컬 PC 운영이면 Windows 작업 스케줄러로 **밤 9:30 자동 기상 / 새벽 6:00 자동 절전**을 걸어 둔다.

### 6단계 10. 대시보드·외부 공개

- `streamlit run`으로 대시보드를 띄운다(기본 8501 포트).
- 외부에서 접속하려면 **Cloudflare Tunnel**로 도메인을 localhost:8501에 연결한다. (포트를 직접 여는 것보다 안전)

### 6단계 11. 백업·운영 점검

- 매일 새벽 `pg_dump` → gzip → (클라우드면 S3) 백업. 최근 며칠분만 보관.
- 디스크 사용량을 주기적으로 확인한다. (대용량 테이블 때문에 디스크가 자주 위기였다 — 6월에 완료한 `morphemes` 신문 샘플링으로 89GB→28GB까지 줄여 이 압박을 크게 덜었다.)
